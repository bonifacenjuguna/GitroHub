## ✨ What is GitroHub?

**GitroHub** is a self-hosted, private Telegram bot that brings your **entire GitHub workflow** into Telegram. No browser, no app switching, no friction.

It lets you create repositories, commit code, upload files, browse your file tree, manage branches, view stats, handle issues and releases, and automate development workflows — all from a single Telegram chat, on any device.

Built using the **GitHub REST API v3** and **Telegram Bot API**, it is optimized for speed, automation, and developer efficiency. It connects via **OAuth 2.0**, encrypts your credentials with **AES-256-GCM**, and gives you a full button-driven interactive UI — no commands to memorize.

> [!NOTE]
> GitroHub is **private by design** — only the `TELEGRAM_ADMIN_ID` you configure can use it. Everyone else sees a custom private access message. It is your personal GitHub remote control, not a public service.

## 🏗️ Architecture

```
┌──────────────────────────────────────────────────────────┐
│                      Telegram User                       │
│              (messages + inline button taps)             │
└────────────────────────┬─────────────────────────────────┘
                         │  HTTPS
                         ▼
┌──────────────────────────────────────────────────────────┐
│                  Telegram Bot API                        │
│              (webhook on Railway / polling)              │
└────────────────────────┬─────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────┐
│                      main.py                             │
│  ┌──────────────────────────────────────────────────┐   │
│  │  CommandHandler    → /start /repos /browse ...   │   │
│  │  CallbackHandler   → button taps & menus         │   │
│  │  MessageHandler    → text state machine           │   │
│  └──────┬─────────────┬──────────────┬─────────────┘   │
└─────────┼─────────────┼──────────────┼──────────────────┘
          │             │              │
    handlers/      handlers/     handlers/   handlers/ ...
    auth.py        repos.py      files.py    upload.py
          │
          ▼
  utils/github_helper.py          utils/encryption.py
  • PyGithub client factory       • AES-256-GCM encrypt
  • Rate limit handling           • Decrypt on token use
  • Formatters & error map
          │
          ▼
    GitHub REST API v3            PostgreSQL (Railway)
    api.github.com                • Sessions (encrypted)
                                  • State machine
                                  • Repos, aliases
                                  • Settings, paths
```

**OAuth 2.0 Flow:**
```
User taps "🔗 Connect GitHub Account"  ←  URL link button ↗
        ↓
github.com/login/oauth/authorize?client_id=...&state=<csrf>
        ↓  User approves
GET /auth/github/callback?code=...&state=<csrf>
        ↓
  ✔ Validate CSRF state token
  ✔ Exchange code → GitHub access token
  ✔ Encrypt token (AES-256-GCM)
  ✔ Store ciphertext → PostgreSQL
        ↓
Bot: "✅ Connected as @username!"
```

## 🗄️ Database Schema

<details>
<summary><b>Click to view full annotated schema</b></summary>

```sql
-- One row per GitHub account per user (supports multi-account)
CREATE TABLE sessions (
    id              SERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL,
    github_username TEXT NOT NULL,
    access_token    TEXT NOT NULL,        -- AES-256-GCM encrypted
    active_repo     TEXT,                 -- "user/repo"
    active_branch   TEXT DEFAULT 'main',
    is_active       BOOLEAN DEFAULT TRUE, -- currently selected account
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);

-- Tracks what the bot is waiting for from the user
CREATE TABLE user_states (
    telegram_id     BIGINT PRIMARY KEY,
    state           TEXT,                 -- e.g. "awaiting_commit_message"
    state_data      JSONB DEFAULT '{}',   -- context payload (files, paths...)
    updated_at      TIMESTAMP DEFAULT NOW()
);

-- Pinned and recently accessed repos
CREATE TABLE user_repos (
    id              SERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL,
    repo_full_name  TEXT NOT NULL,
    is_pinned       BOOLEAN DEFAULT FALSE,
    last_used_at    TIMESTAMP DEFAULT NOW()
);

-- Custom command shortcuts
CREATE TABLE aliases (
    id              SERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL,
    alias           TEXT NOT NULL,        -- "deploy"
    target          TEXT NOT NULL,        -- "/upload dist/app.js main"
    created_at      TIMESTAMP DEFAULT NOW()
);

-- Quick-jump file path shortcuts
CREATE TABLE saved_paths (
    id              SERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL,
    label           TEXT NOT NULL,        -- "config"
    path            TEXT NOT NULL,        -- "src/config/settings.py"
    repo            TEXT NOT NULL
);

-- Commit message fill-in templates
CREATE TABLE commit_templates (
    id              SERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL,
    label           TEXT NOT NULL,        -- "hotfix"
    template        TEXT NOT NULL         -- "fix: {description}"
);

-- Per-user preferences
CREATE TABLE user_settings (
    telegram_id     BIGINT PRIMARY KEY,
    private_message TEXT DEFAULT 'This bot is private.',
    auto_commit_msg BOOLEAN DEFAULT TRUE,
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

> All tables are created automatically by `init_db()` on bot startup. No manual SQL needed.

</details>

## ⚙️ Setup & Deployment

### 📋 Prerequisites

| Requirement | Notes |
|-------------|-------|
| GitHub account | Free or paid |
| Telegram account | Any |
| [Railway.app](https://railway.app) account | Free tier works |
| Python 3.11+ | Local dev only |

---

### Step 1 — Create Your Telegram Bot

1. Message **[@BotFather](https://t.me/BotFather)** → `/newbot`
2. Set a **name** (`GitroHub`) and **username** (`@MyGitroHubBot`)
3. Copy the **Bot Token**: `1234567890:ABCdefGHIjklMNOpqrsTUVwxyz`

> [!WARNING]
> Never commit your bot token. It gives full control over the bot.

---

### Step 2 — Create GitHub OAuth App

Go to **GitHub → Settings → Developer settings → OAuth Apps → New OAuth App**

| Field | Value |
|-------|-------|
| Application name | `GitroHub` |
| Homepage URL | `https://your-app.railway.app` |
| Callback URL | `https://your-app.railway.app/auth/github/callback` |

Copy **Client ID** and generate + copy **Client Secret**.

> [!WARNING]
> Client secret shown only once — copy it immediately.

---

### Step 3 — Generate Encryption Key

```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
# Output: a3f8c2d1...64 hex chars...9f2e1a0b
```

> [!IMPORTANT]
> Back this up. If lost, all sessions become unreadable and all accounts must re-authenticate.

---

### Step 4 — Get Your Telegram User ID

Message **[@userinfobot](https://t.me/userinfobot)** → it replies with your numeric ID (e.g. `123456789`).

---

### Step 5 — Deploy on Railway

1. Push to a **private** GitHub repo
2. Railway → **New Project** → **Deploy from GitHub**
3. Add **PostgreSQL** plugin (auto-injects `DATABASE_URL`)
4. Add variables:

```env
TELEGRAM_BOT_TOKEN     = from BotFather
TELEGRAM_ADMIN_ID      = your numeric Telegram ID
GITHUB_CLIENT_ID       = from GitHub OAuth App
GITHUB_CLIENT_SECRET   = from GitHub OAuth App
WEBHOOK_URL            = https://your-app.railway.app
AES_ENCRYPTION_KEY     = your 64-char hex key
PORT                   = 8080
```

5. Deploy → watch logs for:
```
✅ Webhook set → https://your-app.railway.app/webhook
🚀 GitroHub v1.2.0 running on port 8080
```

6. Open bot → `/start` → tap **Connect GitHub Account** 🎉

---

### Step 6 — Configure BotFather

```
/setdescription →
🚀 Manage GitHub directly from Telegram.
Create repos, commit code, delete files, and automate
development workflows — all without leaving chat.

/setabouttext →
Manage GitHub from Telegram 🚀
Commit, push and control your workflow ⚙️
@GitroHub
```

Upload logo via `/setuserpic`. Commands auto-register on every startup — don't set them manually.

## 💻 Local Development

```bash
# 1. Clone & install
git clone https://github.com/your-username/gitrohub.git
cd gitrohub
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# 2. Configure
cp .env.example .env
# Fill in all values in .env

# 3. Create local database
createdb gitrohub
# Set DATABASE_URL=postgresql://localhost/gitrohub in .env

# 4. Run in polling mode (no public URL needed for updates)
WEBHOOK_URL="" python3 main.py
```

> [!NOTE]
> For the OAuth callback during local dev, use [ngrok](https://ngrok.com): `ngrok http 8080`. Set the ngrok URL as `WEBHOOK_URL` and update your GitHub OAuth App's callback URL temporarily.

## 🔑 Environment Variables

| Variable | Required | What happens if missing |
|----------|:--------:|------------------------|
| `TELEGRAM_BOT_TOKEN` | ✅ | Bot fails to start — no Telegram connection |
| `TELEGRAM_ADMIN_ID` | ✅ | Bot starts but no one can use it |
| `GITHUB_CLIENT_ID` | ✅ | OAuth login fails entirely |
| `GITHUB_CLIENT_SECRET` | ✅ | OAuth token exchange fails |
| `WEBHOOK_URL` | ✅ | Falls back to polling mode (no OAuth callback) |
| `AES_ENCRYPTION_KEY` | ✅ | Tokens can't be encrypted — sessions fail |
| `DATABASE_URL` | ✅ | Bot crashes on startup — no DB connection |
| `PORT` | ✅ | Defaults to `8080` — Railway may complain |
| `GITHUB_REDIRECT_URI` | ⬜ | Auto-built from `WEBHOOK_URL` |

## 💡 Tips & Tricks

> Hidden power features most users miss.

| 💡 Tip | Details |
|--------|---------|
| **Auto-commit messages** | Don't want to type? GitroHub auto-generates descriptive commit messages based on what files changed. Just tap "Auto-generate". |
| **Reuse old messages** | After any commit, your message is saved. On the next commit, tap "Reuse" to pick from recent messages — great for repetitive workflows. |
| **Pin your main repo** | Use `/projects` → pin your most-used repo. It appears at the top every time, one tap to activate. |
| **Branch memory** | Each GitHub account remembers its own active branch. Switch accounts and your branch switches with it. |
| **Cancel anywhere** | Stuck in a flow? `/cancel` always works — it clears any in-progress state and resets the bot cleanly. |
| **Batch = one commit** | All files in a `/batch` session get committed together in one commit, keeping your history clean. |
| **Mirror for fresh deploys** | Use `/mirror` when you want the repo to exactly match your local ZIP — no stale files left behind. |
| **Update for safe syncs** | Use `/update` when you want to add new files and update changed ones without touching anything else. |
| **Search before browse** | For large repos, `/search filename` is faster than navigating the file browser manually. |
| **Diff before merge** | Always run `/diff branch1 branch2` before `/merge` — see exactly what will change. |

## 🔒 Security

GitroHub handles OAuth tokens with **write access to your GitHub account**. Security is non-negotiable.

| 🛡️ Layer | Protection | Details |
|----------|------------|---------|
| **Encryption at rest** | AES-256-GCM | All GitHub tokens encrypted before DB write. Key never in DB. |
| **Access control** | Single-owner | Only `TELEGRAM_ADMIN_ID` can use the bot. All others get a private message. |
| **No hardcoded secrets** | Env vars only | Zero credentials in source code. All loaded from environment. |
| **Transport** | HTTPS only | Railway enforces SSL. HTTP rejected everywhere. |
| **CSRF** | State token | Unique `state` per OAuth flow. Validated before token exchange. |
| **ZIP safety** | Path sanitization | All ZIP paths sanitized — prevents `../../etc/passwd` traversal. |
| **Destructive actions** | Confirmations | Delete and logout require confirmation. Repo delete = 3-step. |
| **Session isolation** | Per-account | Each account is an isolated encrypted session. Switching never leaks tokens. |
| **Token safety** | Never exposed | Token never logged, never in errors, never returned to Telegram in any form. |

> [!IMPORTANT]
> Your GitHub OAuth token exists in plaintext only in RAM for the duration of an API call. At rest, only the AES-256-GCM ciphertext is stored.

## ⚠️ Known Limitations

| Limitation | Details | Workaround |
|-----------|---------|------------|
| GitHub API rate limit | 5,000 req/hr per user | Use `/status` to check remaining; avoid large batch ops near the limit |
| File size cap | 50 MB max (Telegram limit) | Split large uploads; use Git LFS for large assets |
| Binary file editing | Can upload but not edit/preview in chat | Edit locally, re-upload via `/upload` |
| Large repo ZIP mirror | 1 API call per file; 1000+ files may hit rate limit | Use `/update` instead of `/mirror` for large repos |
| No GitHub webhooks | Bot doesn't receive push/PR events automatically | Poll manually with `/stats`, `/log` — webhooks on roadmap |
| Message length | 4,096 char preview limit in Telegram | Large files truncated — use `/download` for full content |
| Single admin | One `TELEGRAM_ADMIN_ID` per instance | Team mode on roadmap |
| GitHub Enterprise | Targets `api.github.com` only | Enterprise support in backlog |

## 🚨 Troubleshooting

<details>
<summary><b>Bot doesn't respond to /start</b></summary>

- Check Railway logs for startup errors
- Verify `TELEGRAM_BOT_TOKEN` is correct and not revoked (BotFather → `/mybots`)
- Confirm `TELEGRAM_ADMIN_ID` exactly matches your numeric Telegram ID
- Check Railway service is deployed and running (not crashed)

</details>

<details>
<summary><b>OAuth callback fails</b></summary>

- `WEBHOOK_URL` must be exact — no trailing slash
- GitHub OAuth App callback must be `https://your-app.railway.app/auth/github/callback`
- A single mismatched character causes GitHub to reject the redirect

</details>

<details>
<summary><b>"❌ Not logged in" after connecting</b></summary>

`AES_ENCRYPTION_KEY` changed after sessions were stored. The ciphertext is now unreadable.

Fix: restore the original key in Railway. If lost: delete all rows in `sessions` table → re-login.

</details>

<details>
<summary><b>Database errors on startup</b></summary>

- Add the PostgreSQL plugin in Railway → `DATABASE_URL` is auto-injected
- `init_db()` creates all tables automatically — no manual SQL needed

</details>

<details>
<summary><b>Webhook not receiving updates</b></summary>

Railway changes URLs on re-deploy without a custom domain. When URL changes:
1. Update `WEBHOOK_URL` env var
2. Update GitHub OAuth App callback URL
3. Re-deploy

</details>

<details>
<summary><b>ZIP upload skips files</b></summary>

- `/mirror` → replaces everything to match the ZIP
- `/update` → adds new + modifies changed. Unchanged = skipped by design. Deleted from ZIP ≠ deleted from repo
- Files over 50 MB in the ZIP are silently skipped (Telegram limit)

</details>

## ❓ FAQ

<details>
<summary><b>Is GitroHub open source?</b></summary>

It's provided as a self-hostable package for personal use. See [License](#️-license).

</details>

<details>
<summary><b>Can multiple people use the same bot?</b></summary>

Only the `TELEGRAM_ADMIN_ID` owner can use it. Team/multi-admin mode is on the [roadmap](#️-roadmap).

</details>

<details>
<summary><b>Does it store my GitHub token?</b></summary>

Yes — AES-256-GCM encrypted, in your own PostgreSQL database that you fully control. Never plaintext.

</details>

<details>
<summary><b>What OAuth scopes does it request?</b></summary>

`repo` (full repo read/write) and `read:user` (username + profile). No email, org admin, or billing access.

</details>

<details>
<summary><b>Can I use it with GitHub Enterprise?</b></summary>

Not yet — targets `api.github.com` only. Enterprise support is in the backlog.

</details>

<details>
<summary><b>What if I lose my AES_ENCRYPTION_KEY?</b></summary>

All sessions become permanently unreadable. Clear the `sessions` table and re-authenticate. Always back the key up outside Railway.

</details>

<details>
<summary><b>Does it work if Railway sleeps?</b></summary>

Free tier may sleep idle services. Upgrade to paid or use always-on setting for uninterrupted operation.

</details>

<details>
<summary><b>Can I run it without Railway?</b></summary>

Yes — any server with Python 3.11+ and PostgreSQL works. VPS, Heroku, Render, Fly.io, local machine. See [Local Development](#-local-development).

</details>

<details>
<summary><b>Is my data safe if someone finds the bot username?</b></summary>

Yes. Non-owners only see the custom private access message. No commands, no repo data, no GitHub access.

</details>

<details>
<summary><b>How fast is the bot?</b></summary>

Most operations complete in 1–3 seconds. GitHub API calls (file reads, commits, stats) depend on GitHub's response time. Batch commits with many files take proportionally longer.

</details>

## 📘 Glossary

| Term | Meaning |
|------|---------|
| **AES-256-GCM** | Advanced Encryption Standard with 256-bit key in Galois/Counter Mode — authenticated encryption used to store your GitHub token |
| **OAuth 2.0** | The authorization protocol GitroHub uses to connect to GitHub without storing your password |
| **Webhook** | An HTTPS endpoint Railway hosts that Telegram sends bot updates to |
| **Polling** | Alternative to webhook — the bot asks Telegram "any new messages?" repeatedly (used in local dev) |
| **Active repo** | The repo GitroHub currently operates on — set via `/use` or by tapping a repo |
| **Active branch** | The branch GitroHub commits to — set via `/switch` |
| **Mirror mode** | ZIP upload that makes the repo exactly match the ZIP — replaces all content |
| **Update mode** | ZIP upload that only adds new and modifies changed files — nothing is deleted |
| **CSRF token** | A one-time random `state` value sent with OAuth to prevent cross-site request forgery |
| **State machine** | GitroHub's conversation tracker — remembers what it's waiting for (e.g. a commit message) |
| **PTB** | python-telegram-bot — the async library powering GitroHub's Telegram integration |
| **Rate limit** | GitHub allows 5,000 API requests per hour per authenticated user |
| **Soft reset** | Undo a commit but keep the file changes staged (reversible) |
| **Hard reset** | Roll back to a commit, discarding all changes after it (irreversible without backup) |

## 🔒 Privacy Policy

**What GitroHub stores:**
- Your Telegram user ID (to verify you're the owner)
- Your GitHub username (for display)
- Your GitHub OAuth token (AES-256-GCM encrypted)
- Your active repo and branch preferences
- Your pinned repos, saved paths, aliases, and templates

**What GitroHub does NOT store:**
- Your GitHub password (OAuth is used — no password ever touches the bot)
- Your email address
- Any file contents (files are sent to GitHub directly — not stored by the bot)
- Any commit history beyond what GitHub itself stores
- Any data from people who message the bot and are not the owner

**Data retention:**
- All data stays in your own Railway PostgreSQL database
- You can delete all your data at any time by clearing the database
- Disconnecting an account (`/logout`) removes that session from the database

**Third parties:**
- GitroHub communicates only with Telegram and GitHub APIs
- No analytics, no tracking, no advertising, no third-party data sharing
